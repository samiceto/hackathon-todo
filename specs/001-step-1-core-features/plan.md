# Implementation Plan: Step 1 - Core Todo Features

**Branch**: `001-step-1-core-features` | **Date**: 2025-12-31 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-step-1-core-features/spec.md`

## Summary

Implement an in-memory Python console todo application with 5 basic CRUD operations (Add, View, Mark Complete, Update, Delete). The application uses Python 3.13+ with stdlib-only dependencies, menu-driven interface, and clean architecture following TDD principles. Storage is session-based using dictionaries and dataclasses, with an abstracted storage layer to facilitate future database migration in Step 2.

**Technical Approach**: Four-module architecture (models, storage, ui, main) with separation of concerns, comprehensive input validation, and user-friendly error handling. All code generated via Claude Code from specifications following spec-driven development workflow.

## Phase 0: Research & Decisions

**Status**: ✅ COMPLETE

**Output**: `design/research.md` (created)

### Key Decisions Made

1. **Data Structures**: Python dict + dataclass for O(1) lookup and type safety
2. **Interface Pattern**: Menu-driven with numbered options (1-6)
3. **Validation Strategy**: Point-of-entry validation with retry loops
4. **Display Formatting**: Manual f-string formatting with column alignment
5. **Module Structure**: 4 modules (models, storage, ui, main)
6. **ID Generation**: Simple integer counter starting at 1
7. **Error Handling**: Defensive programming, no exception propagation to user
8. **Testing Strategy**: pytest with fixtures for isolation
9. **Project Setup**: UV with pyproject.toml configuration
10. **Interrupt Handling**: try/except KeyboardInterrupt in main loop

**Reference**: See `design/research.md` for detailed rationale and implementation patterns.

### Data Model

**Entity**: Task
- **Attributes**: id (int), title (str), description (str), completed (bool), created_at (datetime)
- **Validation**: Non-empty title, auto-generated ID, timestamps
- **Operations**: create, read, update, toggle_complete, delete

**Storage**: TaskStorage class
- **Interface**: add(), get(), get_all(), update(), toggle_complete(), delete(), count()
- **Implementation**: Dict[int, Task] with integer counter
- **Guarantees**: Sequential IDs, sorted output, idempotent operations

**Reference**: See `design/data-model.md` for complete specification and test cases.

   - Add database session management
   - Keep same interface (add, get, update, etc.)

   - Convert dataclass to SQLModel
   - Add database table configuration
   - Preserve validation logic
